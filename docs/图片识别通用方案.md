# Sakura AI 通用文档图片识别方案（OCR/视觉/混合路由）

## 1. 背景与问题定义
目前系统在“需求分析”流程中主要依赖文本抽取：
- 前端 `src/utils/fileReader.ts`：PDF 用 `pdfjs-dist` 的 `page.getTextContent()` 抽取文本；若检测到图片仅提示“图片内容无法提取”，不做 OCR/视觉识别。
- 后端 `server/services/analysisService.ts`：生成需求文档只使用 `text` 作为模型输入（未把图片/图片识别结果作为多模态输入）。
- 因此当文档包含“图片中的文字/截图/图表”时，模型只能看到抽取到的文本，无法理解图片内容。

## 2. 目标
实现一个更通用的“文档理解管线”，覆盖至少以下文件类型：
- PDF（含文本层 / 扫描版 / 嵌入图片）
- DOCX
- HTML/Markdown/TXT（含 `<img>`、dataURI、或图片引用）

统一输出给 AI 的“上下文”包括：
- 文本上下文（文本层抽取结果）
- 图片识别上下文（OCR 或视觉描述结果，转换为可读文本块并附带引用信息）

## 3. 非目标（明确范围）
- 不强求还原文档的全部版式
- 不做“图片像素级渲染回显”
- 不保证对手写体/极低清晰度图片达到满准确率（需降级策略）

## 4. 总体方案概览
### 4.1 组件分层
1) 文档解析层（按文件类型抽取）
- 输入：上传文件 `File`
- 输出（统一结构）：
  - `text: string`
  - `images: ImageAsset[]`（图片资产清单，可包含页码、坐标、或来源信息）

2) 图片处理层（图片识别）
- OCR：从图片中提取文字（用于扫描件/截图含字）
- 视觉描述/图表解读：从图片中提取结构化含义（用于流程图/架构图/表格图片）
- 得到统一结果：`imageCaptions[]`（可含 OCR 文本）

3) 上下文组装层（喂给模型）
- 把 `text` 和每张图片识别结果按块追加到 prompt 中
- 加入引用信息（例如“第 X 页图片 #Y 的 OCR 结果”）

4) 成本与策略控制层
- 路由选择（OCR/视觉/混合）
- 限制图片数量、分辨率、输出长度
- 缓存与去重

### 4.2 混合路由（推荐）
推荐实现“路由规则”，优先用低成本 OCR 补足文字，再用视觉补语义：
- 若检测到“扫描版/PDF 文本量很少” -> OCR
- 若图片类型更可能是“图表/结构” -> 视觉描述
- 若无法判断 -> OCR 优先，必要时再加少量视觉补充

## 5. 路由与判定规则（示例）
路由输入来自现有/新增信号：
- `isScannedPdf`：扫描版PDF判定（你们已有类似逻辑：文本量太少）
- `hasImages`：是否包含图片（你们已有）
- `textQuality`：文本抽取质量指标（例如字符数、页内文本密度、行数等）
- `imageCandidatesCount`：图片候选数量（从提取图片资产后可得）

建议规则（可调参）：
- 规则 A：`isScannedPdf === true` -> OCR（优先对每页做 OCR，最多 N 页）
- 规则 B：`hasImages === true && textQuality >= 阈值` -> OCR（只对关键页/关键图片做 OCR）
- 规则 C：`hasImages === true && textQuality < 阈值` -> 混合（OCR + 视觉取少量图）
- 规则 D：若明确是“流程图/架构图/表格图”（可通过图片尺寸比例、是否存在明显表格线特征、或基于少量先验分类）-> 视觉描述优先

## 6. 图片识别输出规范（关键：避免 token 爆炸）
无论文档/图片类型，最终都要输出成“短且结构化”的文本块，供 AI 作为上下文。

建议统一输出格式（示例）：

1) 文本块（来自文本层抽取）
- `=== Document Text ===`
- 放 `text`（可分段、可截断）

2) OCR 块（图片 -> OCR）
- `=== Image #i OCR (page p) ===`
- 输出：
  - `OCR_TEXT: ...`（限制最大字符数，例如 1500-3000 字符/张）
  - `NOTES:`（可选，标注不确定性或识别质量评分）

3) 视觉描述块（图片 -> 图理解）
- `=== Image #i Visual Summary (page p) ===`
- 输出限制：
  - `SUMMARY:` 3-8 行
  - `ENTITIES:` 识别到的关键对象/模块列表（如适用）
  - `RELATIONS:` 关键关系/流程（如适用）
  - `TABLES:` 表格关键字段与摘要（如适用）

## 7. Token/成本控制策略（落地）
### 7.1 为什么会贵
- OCR：token主要来自 OCR 输出文本长度；但 OCR 本身不一定要消耗 LLM token（取决于你如何接入）。
- 视觉模型：通常比 OCR 更贵，且输入侧可能按图片数量/分辨率/patch 计费（口径因供应商而异），总体趋势是显著高于文本调用。

### 7.2 控制手段（必须做）
1) 图片数量上限
- 每个文档最多处理 `MAX_OCR_IMAGES` 张或 `MAX_OCR_PAGES` 页
- 视觉模型最多处理 `MAX_VISION_IMAGES` 张（通常远小于 OCR）

2) 输出长度上限
- 每张图 OCR 输出字符数上限 `OCR_CHAR_LIMIT`
- 每张图视觉描述输出字符数上限 `VISION_CHAR_LIMIT`
- 超出则截断并保留“关键信息优先”的策略（例如先保留疑似标题/表头/数字字段）

3) 上下文裁剪与摘要
- 当 OCR/视觉块总长度超过 `CONTEXT_CHAR_BUDGET`：
  - 优先保留：与需求相关的段落（标题、步骤、验收标准、字段名等）
  - 其余块做摘要或丢弃

4) 先 OCR 后视觉（混合）
- 大多数“图片=文字”场景用 OCR
- 只有少量“结构信息缺失”场景才用视觉补充

5) 缓存与去重
- 按文档 hash + 页码/图片 hash 缓存 OCR/视觉结果
- 重复上传或多次生成同一需求时直接复用

### 7.3 成本评估方法（建议）
对每次生成记录以下数据（写入日志或数据库）：
- `imagesFoundCount`
- `ocrProcessedCount`
- `visionProcessedCount`
- `ocrTotalChars`
- `visionTotalChars`
- 模型调用的 `promptTokens` / `completionTokens` / `totalTokens`（从供应商响应的 usage 里取）
- 总耗时 `extractTime + ocrTime + visionTime + llmTime`

用这些数据做 A/B 测试最有效：
- baseline：当前只文本抽取
- MVP：OCR优先（少量视觉）
- full：混合路由（动态触发视觉）

## 8. 安全、合规与数据最小化
- 若使用外部 OCR/视觉服务：
  - 只上传必要的图片区域（尽量裁剪到包含内容的区域）
  - 对图片中的敏感字段做脱敏（可选）
  - 配置日志级别，避免把原始图片或 OCR 全文写入过多日志
- 对“个人隐私/合同/账号”等文档：
  - 提供开关：禁止外部识别、仅本地文本层（或仅允许在用户确认后上传）

## 9. 失败模式与降级策略
1) OCR 质量差
- 降级：只保留识别出的数字/字段名/标题行
- 提示：将“识别置信度低”反馈给用户，并建议上传更清晰版本或提供文字补充

2) 视觉模型输出过长
- 限制输出模板并强制截断
- 对“长图”先裁剪 ROI（表格/标题区域优先）

3) OCR/视觉超时
- 超时则跳过该页/该图，继续生成文本层结果
- 同时在 UI 中提示“部分图片未识别”

## 10. 最小可行版本（MVP）建议实现路线
### 10.1 MVP 功能清单
- PDF：
  - 继续抽取文本层
  - 对“扫描版PDF或文本量不足”页执行 OCR
- DOCX：
  - 保留现有文字抽取
  - 若检测到内嵌图片，先只做 OCR（或先提示用户需要补充图片文字）
- HTML/MD：
  - 提取 `<img>`（仅当能拿到图片数据或可下载资源时）
  - 对图片做 OCR（可配置）

### 10.2 MVP 里程碑
Milestone 1：统一数据结构（`text + images[]` + `imageRecognitions[]`）
Milestone 2：OCR 接入与输出块模板
Milestone 3：上下文组装到现有 `generateRequirementDoc(text)` 的 prompt 中（或新增函数支持 `context`）
Milestone 4：成本控制（上限、截断、缓存）
Milestone 5：UI 提示与可观测性（显示识别是否完成、识别来源）

## 11. 与你们现有代码的对接点（建议）
- 前端 `src/utils/fileReader.ts`
  - 保留文本抽取
  - 新增返回 `images` 或 `imagePlaceholders`（至少返回“每张图的来源信息”）
- 后端 `server/routes/analysis.ts`、`server/services/analysisService.ts`
  - 接收 `images` 或接收“图片识别结果文本块”
  - 让 prompt 使用：`text + ocr/vision块`
- UI（如 `src/pages/RequirementAnalysis.tsx`）
  - 在预览区显示“图片识别状态/警告”
  - 在生成区提示“已识别 X 张图片，追加到上下文”以便用户理解效果

## 12. 待确认问题
1) 你们更希望识别的图片内容是：
   - A：图片里的文字（OCR为主）
   - B：流程图/架构图/表格语义（视觉描述为主）
2) OCR/视觉是否允许调用外部云服务（涉及合规与成本），还是必须本地离线 OCR？
3) 期望的最大识别耗时（例如每份文档 <= 60s）和最大费用预算（例如每份 <= 某阈值）。
